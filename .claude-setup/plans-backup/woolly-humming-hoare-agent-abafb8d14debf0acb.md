## 1. Регистрация программы для ЭВМ в Роспатенте (ФИПС)

### 1.1 Зачем регистрировать

- Свидетельство о государственной регистрации -- официальное подтверждение авторства и прав
- Обязательно для включения в реестр российского ПО Минцифры
- Усиливает позицию в суде при спорах об авторстве
- Требуется Сколково для подтверждения РИД
- Свидетельство действует в течение всего срока авторского права (жизнь автора + 70 лет)

### 1.2 Полная процедура регистрации

**Этап 1: Подготовка документов**

Необходимый пакет:
1. **Заявление** -- форма РП (для программ ЭВМ). Заполняется на сайте ФИПС или через Госуслуги
2. **Реферат** -- до 900 знаков, описание программы
3. **Депонируемые материалы** -- исходный код (полный или фрагменты) + скриншоты
4. **Документ об уплате пошлины** -- квитанция или платёжное поручение
5. **Согласие автора** -- на указание в реестре (если автор не заявитель)
6. **Согласие на обработку персональных данных** -- для физлиц-авторов
7. **Доверенность** -- если подаёт представитель (нотариальная)

**Этап 2: Подача заявки**

Три способа подачи:
- **Через портал ФИПС** (https://new.fips.ru) -- нужна КЭП (квалифицированная электронная подпись), скидка 30% на пошлину
- **Через Госуслуги** -- нужна КЭП, скидка 30%
- **Лично/почтой** в Роспатент: Москва, Бережковская наб., д. 30, корп. 1

**Этап 3: Рассмотрение (до 62 рабочих дней)**
- 10 рабочих дней -- проверка уплаты пошлины
- 10 рабочих дней -- проверка соответствия документов
- 40 рабочих дней -- экспертиза (формальная, не по существу!)
- 2 рабочих дня -- внесение в реестр после положительного решения

На практике: обычно 2-3 месяца, часто быстрее.

**Этап 4: Результат**
- Свидетельство о государственной регистрации программы для ЭВМ
- Публикация в официальном бюллетене Роспатента

### 1.3 Требования к реферату (КРИТИЧНО -- до 900 знаков)

**Обязательная структура:**
1. Полное название программы (как в заявлении)
2. Назначение программы
3. Область применения
4. Функциональные возможности (основные)
5. Язык программирования (на языке оригинала: Python, TypeScript и т.д.)
6. Объём программы в байтах (или кратных единицах: КБ, МБ)
7. Тип ЭВМ (если есть ограничения по платформе)

**Пример реферата для платформы производства:**

```
Программа «[Название]» предназначена для автоматизации управления производственными
процессами предприятий пищевой промышленности с применением методов искусственного
интеллекта. Область применения: производство биологически активных добавок, продуктов
функционального питания. Функциональные возможности: планирование производства,
управление рецептурами, контроль качества сырья и готовой продукции, прогнозирование
спроса на основе ML-моделей, автоматическая генерация производственной документации,
интеграция с маркетплейсами (Wildberries, Ozon). Язык программирования: Python 3.11,
TypeScript 5.x. Объём программы: 15,4 МБ.
```

**Важно:**
- Все сокращения расшифровывать
- Печать через 1,5 интервала, заглавные буквы не менее 2,1 мм
- НЕ включать коммерческую информацию, цены, маркетинг
- Реферат публикуется ОТКРЫТО в бюллетене

### 1.4 Требования к депонируемым материалам

**Формат:** PDF/A (обязательно)

**Содержание:**
- Исходный текст программы -- полный или фрагменты
- При подаче фрагментов: сквозная нумерация страниц + указание положения в полном коде
- Допускаются скриншоты интерфейса
- Допускаются подготовительные материалы (блок-схемы, диаграммы)

**Титульный лист депонируемых материалов (обязателен):**
- Название программы
- ФИО/наименование правообладателя
- ФИО всех авторов (если не отказались от указания)

**Объём:** формально не ограничен, но Роспатент рекомендует не более 50 страниц.

**Что включать в листинг:**
- Первые и последние 25 страниц кода (если код большой)
- Или наиболее характерные фрагменты, идентифицирующие программу
- Ключевые модули с уникальной бизнес-логикой
- НЕ включать стандартные библиотеки и фреймворки
- НЕ включать секреты, API-ключи, пароли (очевидно)

### 1.5 Стоимость

| Вид пошлины | Физлицо | Юрлицо | С электронной подачей (скидка 30%) |
|---|---|---|---|
| Регистрация программы ЭВМ | 3 000 руб. | 4 500 руб. | 2 100 / 3 150 руб. |
| Внесение изменений в реестр | 1 200 руб. | 2 500 руб. | 840 / 1 750 руб. |

### 1.6 Частые ошибки при регистрации

1. Реферат превышает 900 знаков -- приостановка
2. Не указан язык программирования или объём -- запрос дополнительных сведений
3. Нет согласия автора -- отказ
4. Депонируемые материалы не в PDF/A -- запрос переоформления
5. Неоплаченная или неправильно оплаченная пошлина -- приостановка
6. Название в заявлении не совпадает с названием в реферате

## 2. ГОСТы на программное обеспечение

### 2.1 Какие ГОСТы применимы к проекту (платформа для производства)

| ГОСТ | Название | Применимость | Обязательность |
|---|---|---|---|
| **ГОСТ 19 серия (ЕСПД)** | Единая система программной документации | Для документирования ПО | Добровольная, но рекомендуется для Сколково |
| **ГОСТ 34 серия** | Автоматизированные системы | Для документирования АС целиком | Добровольная, применяется при госзаказах |
| **ГОСТ Р ИСО/МЭК 12207-2010** | Процессы жизненного цикла ПО | Для описания процессов разработки | Добровольная, хороша для отчётов Сколково |
| **ГОСТ Р 56939-2024** | Безопасная разработка ПО | Для подтверждения безопасности | Обязательна при сертификации ФСТЭК |

### 2.2 ГОСТ 34 серия (Автоматизированные системы)

**Применимость:** если платформа рассматривается как автоматизированная система управления производством.

**Ключевые стандарты:**
- **ГОСТ 34.201-2020** -- Виды, комплектность и обозначения документов при создании АС
- **ГОСТ 34.601-90** -- Стадии создания АС
- **ГОСТ 34.602-2020** -- Техническое задание на создание АС
- **ГОСТ 34.603-92** -- Виды испытаний АС

**Стадии по ГОСТ 34.601-90:**
1. Формирование требований к АС
2. Разработка концепции АС
3. Техническое задание
4. Эскизный проект
5. Технический проект
6. Рабочая документация
7. Ввод в действие
8. Сопровождение АС

**Документы по ГОСТ 34.201 (основные):**
- Техническое задание (ТЗ)
- Пояснительная записка к техническому проекту
- Описание автоматизируемых функций
- Описание информационного обеспечения
- Описание программного обеспечения
- Руководство пользователя
- Программа и методика испытаний

### 2.3 ГОСТ Р ИСО/МЭК 12207-2010 (Жизненный цикл ПО)

**Применимость:** для описания процессов разработки в отчётах Сколково.

**Основные группы процессов:**
- **Основные:** приобретение, поставка, разработка, эксплуатация, сопровождение
- **Вспомогательные:** документирование, управление конфигурацией, обеспечение качества, верификация, валидация, аудит
- **Организационные:** управление, создание инфраструктуры, усовершенствование, обучение

**Для Сколково важно:** стандарт описывает процессы, которые можно напрямую отразить в годовом отчёте как подтверждение R&D деятельности.

### 2.4 ГОСТ Р 56939-2024 (Безопасная разработка)

**Применимость:** обязательна при сертификации ФСТЭК; добровольна, но даёт серьёзные очки для Сколково.

**25 процессов**, из которых 8 новых:
- Планирование процессов безопасной разработки
- Моделирование угроз и анализ поверхности атаки
- Формирование и поддержание правил кодирования
- Композиционный анализ (отслеживание уязвимостей в зависимостях)
- Управление секретами и ключами подписи кода
- Обеспечение целостности кода
- Проверка цепочек поставок (supply chain security)
- Нефункциональное тестирование
- Безопасность при эксплуатации и выводе ПО из эксплуатации

**Практические требования:**
- Запрет хранения секретов в открытом виде в коде
- Обязательное документирование всех процессов
- IT-инфраструктура для автоматизации проверок (CI/CD)
- Формирование культуры безопасной разработки

### 3.4 Автогенерация документации из кода

**Готовых инструментов для генерации документации по ГОСТ 19 нет.** Однако можно выстроить пайплайн:

**Стратегия: docstrings -> Sphinx -> ГОСТ-шаблоны**

1. **Исходный код:** писать docstrings на русском языке в формате Google/NumPy
2. **Sphinx + autodoc:** генерировать API-документацию из docstrings
3. **Jinja2 шаблоны:** создать шаблоны документов по ГОСТ 19 с placeholder-ами
4. **AI-обработка:** использовать Claude/GPT для преобразования Sphinx-вывода в формат ГОСТ
5. **LaTeX/DOCX:** финальное оформление в соответствии с ГОСТ 19.106-78

**Рекомендуемый формат docstring для совместимости с ГОСТ:**

```python
def calculate_production_plan(
    recipes: list[Recipe],
    demand_forecast: DemandForecast,
    inventory: Inventory
) -> ProductionPlan:
    """Расчёт производственного плана на основе прогноза спроса.

    Назначение:
        Формирование оптимального плана производства с учётом
        доступного сырья, производственных мощностей и прогноза спроса.

    Алгоритм:
        1. Анализ прогноза спроса по каждому SKU
        2. Проверка наличия сырья по рецептурам
        3. Оптимизация загрузки производственных линий (линейное программирование)
        4. Формирование календарного плана выпуска

    Входные данные:
        recipes: Список рецептур продукции (см. модель Recipe)
        demand_forecast: Прогноз спроса на период (см. модель DemandForecast)
        inventory: Текущие остатки сырья и готовой продукции

    Выходные данные:
        ProductionPlan: Календарный план производства с указанием
        объёмов выпуска, потребности в сырье и загрузки оборудования

    Ограничения:
        - Горизонт планирования: 1-90 дней
        - Максимум 1000 SKU в одном плане
        - Требует наличия актуальных рецептур в БД

    Автор: Гагауз Д.
    Версия: 1.0.0
    Дата: 2026-03-25
    """
```

## 4. Реестр российского ПО (Минцифры)

### 4.1 Условия включения (актуально на март 2026)

**Требования к правообладателю:**
- Российское юрлицо или ИП
- Доля российского участия (контроля) > 50%
- Не менее 50% голосов принадлежит гражданам РФ, субъектам РФ или муниципальным образованиям (новое с 2026)

**Требования к ПО:**
- Исключительные права принадлежат заявителю на территории всего мира
- ПО зарегистрировано в Роспатенте (свидетельство о регистрации)
- Интерфейс на русском языке
- Разработка и поддержка осуществляются в России
- Выплаты иностранным правообладателям компонентов <= 30% выручки
- Нет ограничений на использование в РФ
- Отсутствие государственной тайны
- Гарантийное обслуживание российской организацией

**Новые требования с 2026:**
- Совместимость с российскими операционными системами (поэтапно с 01.09.2026)
- Первый этап (сентябрь 2026) -- офисное ПО
- Промышленное ПО -- к 2028 году
- Вместо критерия "владения" -- критерий "контроля"

### 4.2 Процедура включения

4. **Экспертиза** -- до 65 рабочих дней

### 5.4 Как фиксируется ноу-хау (коммерческая тайна)

**Для введения режима коммерческой тайны нужно:**

1. **Приказ об установлении режима коммерческой тайны** -- подписывает гендиректор
2. **Перечень сведений, составляющих коммерческую тайну** -- конкретный список (алгоритмы, модели, метрики, формулы)
3. **Положение о коммерческой тайне** -- внутренний документ компании
4. **NDA с каждым сотрудником** -- обязательство о неразглашении
5. **NDA с контрагентами** -- при передаче информации
6. **Маркировка документов** -- гриф "Коммерческая тайна" + реквизиты обладателя
7. **Журнал учёта доступа** -- кто получил доступ к каким сведениям

**Преимущества ноу-хау vs регистрация:**
- Не требует раскрытия информации (в отличие от реферата в Роспатенте)
- Действует бессрочно (пока соблюдается режим)
- Можно защитить алгоритмы, метрики, бизнес-логику (то, что не патентуется)
- Не нужна пошлина

**Недостатки:**
- Нет государственного свидетельства
- Сложно доказать в суде
- Теряется при разглашении

**Рекомендация:** для Сколково -- КОМБИНИРОВАТЬ оба подхода:
- Регистрация ПО в Роспатенте (для свидетельства и реестра)
- Ноу-хау для алгоритмов ML, бизнес-логики, уникальных методов (не раскрываемых в реферате)

Скрипт `scripts/check_copyright.py`:

```python
#!/usr/bin/env python3
"""Проверка наличия copyright header в файлах Python."""
import sys

REQUIRED_LINE = "# Copyright (c)"
COMPANY = "Арагант"

def check_file(filepath: str) -> bool:
    with open(filepath) as f:
        content = f.read()
    if REQUIRED_LINE not in content:
        print(f"ОШИБКА: Отсутствует copyright header в {filepath}")
        return False
    if COMPANY not in content:
        print(f"ПРЕДУПРЕЖДЕНИЕ: Не указано ООО «Арагант» в {filepath}")
    return True

if __name__ == "__main__":
    failed = False
    for filepath in sys.argv[1:]:
        if not check_file(filepath):
            failed = True
    sys.exit(1 if failed else 0)
```
