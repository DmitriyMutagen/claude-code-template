# Spec-First — Спецификация ДО кода (ОБЯЗАТЕЛЬНО для задач >3 файлов)

## Когда срабатывает
Любая задача затрагивающая >3 файлов ИЛИ новая фича ИЛИ новый API.

## Обязательный порядок (роль: Designer)

### 1. Spec Document
Создать `docs/plans/YYYY-MM-DD-feature-name.md`:
```markdown
# Feature: [Название]

## Что делает (user story)
Как [пользователь] я хочу [действие] чтобы [результат]

## Acceptance Criteria
- [ ] Критерий 1 (конкретный, проверяемый)
- [ ] Критерий 2
- [ ] Критерий 3

## API Contract (если есть)
POST /api/endpoint
Input: { field: type }
Output: { field: type }
Errors: 400, 401, 404, 500

## Data Model (если есть)
Table: name
  - field: type (constraint)

## Edge Cases
- Что если пусто?
- Что если 100K записей?
- Что если API недоступен?

## НЕ-требования (что явно НЕ делаем)
- [explicit scope boundary]

## Оценка
⏱️ X часов (Y агентов параллельно)
```

### 2. Согласование
Показать spec Дмитрию. НЕ начинать код без "ок".

### 3. Только потом код
Код должен соответствовать spec. Если spec неверный — исправь spec, потом код.

## Запрещено
- Писать код >3 файлов без spec
- "Очевидные" фичи без acceptance criteria
- Деплой без проверки acceptance criteria
