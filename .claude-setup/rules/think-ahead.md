# Think Ahead — 10 шагов вперёд (ОБЯЗАТЕЛЬНО)

## При каждом архитектурном решении
- Что будет при ×10 нагрузке? ×100?
- Нужно ли сейчас об этом думать или это преждевременная оптимизация?
- Какой технический долг создаём? Когда он выстрелит?

## При каждой новой фиче
- Какие edge cases? (пустые данные, 100K записей, спецсимволы, Unicode)
- Что сломается в существующем функционале?
- Нужна ли миграция данных?
- Обратная совместимость API?

## При каждом API эндпоинте
- [ ] Авторизация (кто может вызвать?)
- [ ] Rate limiting (от DDoS и злоупотреблений)
- [ ] Валидация входных данных (Pydantic/Zod)
- [ ] Обработка ошибок (не 500, а человеческое сообщение)
- [ ] Документация (OpenAPI автогенерация)
- [ ] Логирование (Sentry + structlog)

## При каждом деплое
- [ ] Health check эндпоинт (/health)
- [ ] Docker healthcheck в compose
- [ ] Rollback план (предыдущий Docker image)
- [ ] Мониторинг (Sentry)
- [ ] Бэкап БД перед миграцией

## При каждом git commit
- Сообщение описывает ЗАЧЕМ, не ЧТО
- Не коммитить .env, секреты, большие бинарники
- Conventional Commits (feat/fix/refactor/docs)
