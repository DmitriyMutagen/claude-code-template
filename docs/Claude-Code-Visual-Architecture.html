Rules (28)

Enforcement rules that govern every aspect of the development process. Blocking rules prevent completion without compliance. Quality rules ensure standards.

🔒
serena-sentry-enforcement BLOCKING
Mandatory Serena activation for code navigation and Sentry check before any bug fix. No exceptions.
✅
verify-before-done BLOCKING
No "done" without real proof: screenshot, curl output, passing test, or Playwright verification.
📁
git-workflow-enforcement BLOCKING
Feature branches only. Never force-push main. Conventional commits. Pre-commit hooks required.
📦
modular-architecture BLOCKING
Domain-driven structure enforced. Import rules: domain cannot import from another domain directly.
🧪
testing-always BLOCKING
Every code change requires tests. Bug fix requires regression test first (TDD). No deploy without green.
🛠
quality-gates QUALITY
Pre-code checklist: read MEMORY.md, check Sentry, run existing tests, activate Serena project.
📄
docs-always QUALITY
Google-style docstrings mandatory. 80% function coverage, 90% class coverage. Interrogate enforced.
📈
coding-standards QUALITY

📊
data-correctness QUALITY

🔗
chain-thinking WORKFLOW

🔎
research-before-code WORKFLOW

spec-first WORKFLOW

think-ahead WORKFLOW

time-estimation WORKFLOW

Decision Framework

CTO Layer (4 Phases)

Every prompt is processed through four phases. Claude acts as CTO, routing decisions top-down: strategy first, then architecture, then code. Never jump straight to implementation.

0
Understanding
Interpret real intent (not literal words). Classify scale: Quick fix (<5 min) / Task (1-3 days) / Project (>3 days). Ask clarifying questions if ambiguous.
1
Strategy
WHY before HOW. Business goal identification. Priority check (top-3?). Cost of inaction. Challenge if wrong direction: "real problem is Y, let's do Z instead."
2
Architecture & Design
Modules, data flow, dependencies, tool selection. 3 variants minimum: Quick / Balanced / Ideal. System impact analysis. Definition of Done before code. Approval required.
3
Implementation + Verification
Break into tasks, execute sequentially. Code -> tests -> verify -> next. If architecture fails -> STOP, return to Phase 2. Tests pass, lint clean, Sentry clear.

Stop Signals

Anti-Patterns

Recognized failure modes that trigger automatic STOP and course correction. These patterns are enforced by hooks and instincts to prevent wasted effort.

⚠
"Let me quickly code this"
STOP. Phase 1 (Strategy) was skipped. Why are we doing this? What metric does it move?
⚠
"Let's also add X while we're at it"
STOP. Scope creep. What's in MVP? What's for later? Finish current task first.
⚠
"We need to rewrite everything"
STOP. What's the minimum change to achieve the goal? Rewrite is almost never the answer.
⚠
"It's obvious, no need for a plan"
STOP. If it takes >1 day, show 3 variants with trade-offs. Nothing is obvious at scale.
⚠
Code without approved plan
STOP. Return to Phase 2. Get architecture approval before writing a single line.
⚠
"Done" without verification
STOP. "Should work" is not done. Provide: test output, curl result, screenshot, or Playwright proof.
