Правила (28)

Правила-гарды контролируют каждый аспект процесса. Блокирующие не дают закрыть задачу без соблюдения.

🔒
serena-sentry-enforcement БЛОК
Обязательная Serena для навигации и Sentry перед любым фиксом. Без исключений.
✅
verify-before-done БЛОК
Никакого "готово" без доказательства: скриншот, curl, тест или Playwright.
📁
git-workflow-enforcement БЛОК
Только feature-ветки. Без force-push main. Conventional commits. Pre-commit обязателен.
📦
modular-architecture БЛОК
Domain-driven структура обязательна. Домен не импортирует из другого домена.
🧪
testing-always БЛОК
Каждое изменение = тесты. Фикс бага = regression-тест первым. Деплой только с зелёными.
🛠
quality-gates КАЧЕСТВО
Чеклист: MEMORY.md, Sentry, тесты, Serena -- до написания кода.
📄
docs-always КАЧЕСТВО
Google-style docstrings. 80% функций, 90% классов. Interrogate включён.
📈
coding-standards КАЧЕСТВО
Строгие типы, async/await, SQLAlchemy 2.0, tenacity retry, таймауты.
📊
data-correctness КАЧЕСТВО
Проверять реальные данные. Сверять БД и фронт. Пустой массив = баг.
🔗
chain-thinking ВОРКФЛОУ

spec-first ВОРКФЛОУ

think-ahead ВОРКФЛОУ

CTO Layer (4 фазы)

Каждый промпт через 4 фазы. Claude = CTO: стратегия->архитектура->код. Никогда сразу в код.

0
Понимание
Интерпретация намерения. Масштаб: мелочь/задача/проект. Уточнения.
1
Стратегия
ЗАЧЕМ до КАК. Бизнес-цель. Top-3? Стоимость бездействия. Челлендж.
2
Архитектура
Модули, data flow, 3 варианта. Влияние. Definition of Done. Согласование.
3
Реализация
Код->тесты->проверка->далее. Не работает = СТОП, к Фазе 2.
